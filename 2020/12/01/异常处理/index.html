异常处理

发表于 2020-12-01 更新于 2020-12-03 分类于 Java
处理错误

在Java中，如果一个方法不能够采用正常的途径完成它的任务，可以通过另外一个路径退出方法。在这种情况下，方法并不返回任何值，而是抛出一个封装了错误信息的对象。需要注意的是，这个方法将会立即退出，并不返回任何正常值。此外，也不会从调用这个方法的代码继续执行，取而代之的是，异常处理机制开始搜索能够处理这种异常情况的异常处理器。

异常分类

在Java语言中，所有的异常都是由Throwable类的类实例，但在下一层又分解为两个分支：Error与Exception。

Error层次描述了Java运行时系统的内部错误和资源耗尽错误。应用程序不应抛出这种错误。

重点关注的是Exception层次，这一层次又分解为两个分支：RuntimeException与其他异常。一般规则是，由编程错误导致的异常属于RuntimeException，如果程序本身没有问题，但由于像I/O错误这类问题导致的异常属于其他异常。

派生于RuntimeException的异常包括以下问题：

  • 错误的强制类型转换
  • 数组访问越界
  • 访问null指针

其他异常主要有：

  • 试图超越文件末尾继续读取数据
  • 试图打开一个不存在的文件
  • 试图根据给定的字符串查找Class对象，而这个字符串表示的类并不存在

Java语言规范将派生于Error类与RuntimeException类的所有异常称为非检查型异常，所有其他异常称为检查型异常。编译器将检查你是否为所有的检查型异常提供了异常处理器。

声明检查型异常

Java中，一个方法必须声明所有可能抛出的检查型异常，而非检查型异常要么在控制之外（ERROR），要么是由从一开始就应该避免的情况所导致的（RuntimeException）。如果你的方法没有声明所有的检查型异常，编译器就会发出一个错误信息。

如果方法中的任何代码抛出了catch子句中没有声明的一个异常类型，那么这个方法就会立即退出。

你可以在程序中捕获异常或者将异常抛出，通常来说，要捕获那些你知道如何处理的异常，而继续传播那些你不知道怎样处理的异常。

但是，如果编写一个方法覆盖超类的方法，而这个超类方法没有抛出异常，你就必须捕获自己代码中出现的每一个检查型异常，不允许在子类的throws说明符中出现超类方法未列出的异常类。

可以在一个try语句块捕获多个异常类型，并对不类型的异常做出不同的处理，要为每个异常类型使用一个单独的catch子句。

使用断言

Java中，给出了3种处理系统错误的机制：

  • 抛出异常
  • 断言
  • 日志

而断言的使用只是在开发与测试阶段，因此，不应该使用断言向程序的其他部分通知发生了可恢复性的错误，或者不应该利用断言与程序用户沟通问题。
